HELP & Tips Built To Be Rugged The goal of this system is to make the process of building a database that can live in two realities, the first being built, stored and navigated from within an application. The other reality is a system of files (records) and folders (tables) that are traditionally organized. In a way filesystems were the first databases, without the benefit of automation for record creation and enforcing namming conventions. This system builds a system of folders and files structure that match what is in the system. The reality is there aren't two databases at work, the structure that lives in storage as files and folders contains the required metadata for the software to make use of in the way it needs to opperate. Moving the database is as easy as zipping the entire folder architecture, expanding it into a new location and launching the application (the Flat File web browser api). At present only Chome and Edge can run using the filespace outside the browser save folder for chaced data and pages. We are working on a solution for a utility that uses the local folder to store the changes the user makes and updates a folder system of the users preference. Regarding Storage - Using Chrome or Edge opens up the data base beyond the broweser storage. Make it look Natural - What you build in the data base is reflected in the file system, even case sensitivity. Don't hesitate to use captilazation and spaces when choosing names for tables and fields, the database can make sense of formatting. - Don't use forwards or backslashes ('/' or '\') as they are used to indicate a directory. - Don't use astrisks ('*') in namming Tables and Fields, they are used as wildcards. What's the deal with the 'slug' field? - The filenames for records have two underscores ("__") in the middle of the filename to separate the 'id' field and the 'slug' field. The 'id' is readOnly and is autogenerated where the 'slug' can be anything the user wants. So if a person's name changes it can be changed in the 'slug' and it wont affect the database from finding the file, since it is only looking at the prefix of the filename (the 'id') and ignores the suffix () - Records in traditional databases are the rows in a file, and the fields are delimited (separated) by a comma, or a tab. In this system records are individual files, fields are individual rows, and are stored in a folder which is the table. I wanted to ensure that renaming things shouldn't break the system, as long as the renaming is done by the database. To make this possible, record filenames that are meant to be human readable need to have filenames that can change and not loose referential integrity. - Shared tables are able to be a Parent to many Children tables. So if you Wanted to create a branching comment thread for example, the original post would be a Shared table, and all the comments that follow would be children and replies to comments would use a link field to point to the specific posted comment or reply. This makes all the comments to a main post be seen as children and working out who was replying to whom is resolved by following the link. Changes as of 4/23/2026 v0.1.2.1
